Overview

- Use Firebase Authentication (client) for signup/sign-in.
- Use server-side verification of Firebase ID tokens via **public keys/JWKS** (no Admin service account).
- No explicit Firebase `uid` validation/storage is required; we link users by the verified ID token email claim.
- On signup: create/upsert a Supabase `users` row (via Prisma) with `status = pending` and `email_verified` from Firebase.
- On sign-in: verify token, update `email_verified`, then allow login only if `status = approved` and `email_verified = true`.
- Create an httpOnly cookie (server-set) to persist the session for protected pages.

```mermaid
flowchart LR
  A[User Signs Up (Firebase client)] --> B[Client calls /api/auth/register]
  B --> C[Server verifies Firebase ID token]
  C --> D[Prisma upsert users row in Supabase]
  D --> E[status=pending (no session set)]

  A2[User Signs In (Firebase client)] --> B2[Client calls /api/auth/session]
  B2 --> C2[Server verifies Firebase ID token]
  C2 --> D2[Prisma update email_verified + check status]
  D2 --> F{approved && email_verified?}
  F -- yes --> G[Server sets httpOnly session cookie]
  F -- no --> H[Return error: NOT_APPROVED or EMAIL_NOT_VERIFIED]

  P[Protected page] --> Q[Server reads cookie + verifies token + checks Prisma]
```

### 1) Dependencies + Prisma setup

- Add dependencies:
  - `prisma`, `@prisma/client`
  - `zod` (request validation)
  - a JWT verifier such as `jose` for public-key ID token verification
- Add `prisma` folder:
  - `prisma/schema.prisma`
  - `prisma/migrations/*` generated by `prisma migrate dev`
- Add minimal helper files:
  - `src/lib/prisma.ts` (or `lib/prisma.ts`) with a PrismaClient singleton

### 2) Prisma `users` table (as requested)

Create `User` model with at least:

- `first_name: string`
- `last_name: string`
- `email_verified: boolean`
- `status: enum` (at least `pending`, `approved`; optionally `rejected`)
  Also include necessary fields:
- `email: string` (unique, used as the auth join key)
- `created_at`, `updated_at`
- Optional but useful:
  - `approved_at: DateTime?`
  - `last_login_at: DateTime?`

### 3) Firebase client + server initialization

- Client Firebase config (safe for browser): `lib/firebase/client.ts`
- Server public-key ID-token verifier: `lib/firebase/verifyIdTokenPublic.ts`
  - Fetch/cache Google public keys/JWKS and verify JWT signature + required claims

### 4) Backend API routes

Add Next.js App Router route handlers:

- `app/api/auth/register/route.ts`
  - `POST` body (validated by Zod): `{ idToken, firstName, lastName }`
  - Verify ID token via public-key JWT verification
  - Upsert Prisma `users` row with:
    - `email`, `first_name`, `last_name`
    - `email_verified` = token `email_verified` claim (derived from decoded ID token)
    - `status` default `pending`
- `app/api/auth/session/route.ts`
  - `POST` body: `{ idToken }`
  - Verify token via public-key JWT verification
  - Upsert/update `users.email_verified`
  - Check `users.status === 'approved'` and `email_verified === true`
  - On success, set httpOnly cookie with the Firebase ID token (or a server-signed session token if you prefer that later)
  - On failure, return structured errors (no secrets)
- `app/api/auth/logout/route.ts`
  - Clear cookie

Security notes in plan:

- Validate all request bodies with Zod.
- Never expose server-side token verification secrets (this approach does not require service-account credentials).
- Cookie is `httpOnly`, `sameSite=Lax`, and only `secure` in production.
- Add a basic IP-based rate limit (best-effort MVP) in both register/session routes.
